# AGENTS.md — 我们的工作方式

这份文件定义了我们在协作中的默认规则和习惯，是项目开发的操作手册。所有内容均源于我们的共同约定，请严格遵守并定期回顾更新。

## 1. 启动流程
每次对话开始时，你必须：
1. 读取 `SOUL.md` 和 `USER.md` 刷新对我的了解。
2. 检查 `memory/YYYY-MM-DD.md`（今日计划）和昨天日志，了解未完成任务。
3. 如果有GitHub上的新Issue或PR，可以提醒我处理。
4. 扫描项目代码中的TODO注释，识别潜在技能缺口（见第4节）。

## 2. 自主操作核心原则
你被授予**最大代码权限**，可以自主修改、重构、优化，但必须严格遵守以下流程：
1. **先理解后修改**：任何改动前，必须完整阅读相关代码文件，理解其功能和上下文。对于不熟悉的技术栈，先学习（见第4.2节）。
2. **评估影响范围**：分析改动可能影响的模块、API、数据库、前端界面。若是核心交易逻辑，必须先在测试环境验证。
3. **做好备份**：所有修改必须在Git分支上进行，提交前确保编译通过、单元测试通过。重要改动必须推送到GitHub远程仓库（即使分支未合并），以防止本地丢失。
4. **提交规范**：Commit message 清晰描述改动内容和原因，关联对应Issue（如果有）。PR描述中说明改动背景、测试情况。
5. **测试先行**：对于新增功能，必须编写对应的单元测试或集成测试。对于优化，需提供性能对比数据。
6. **文档同步**：修改API需更新接口文档；修改配置需更新示例文件。

## 3. 策略监控与自主优化流程

### 3.1 胜率计算逻辑标准化
- **数据源**：从数据库 `trades` 表获取最近30笔实盘交易，按策略ID过滤（策略ID从上下文中获取）。
- **计算方法**：胜率 = 盈利交易数 / 总交易数。
- **缓存**：计算结果存储到Redis，key格式：`strategy:winrate:{id}:{date}`，过期时间1小时，避免重复计算。
- **代码实现**：编写 `internal/calc/win_rate.go`，函数≤50行，并编写单元测试 `calc_win_rate_test.go`。

### 3.2 冷静期校验机制
- **记录**：每日胜率写入Redis（见3.1），保留最近2天数据。
- **校验逻辑**：当收到胜率检查触发信号时，查询近2天的胜率，如果均低于70%，则进入优化流程；否则，记录到 `memory/strategy_watch.md` 并等待明日。
- **避免短期波动**：连续2天低于阈值才触发，确保稳定性。

### 3.3 策略备份自动化
- **触发时机**：进入优化流程后，立即执行备份。
- **备份目录**：`backups/strategies/{strategy_id}/{yyyyMMdd_HHmmss}/`
- **备份内容**：
  - 策略代码：`strategies/active/` 下的所有文件
  - 参数配置：当前使用的配置文件（如 `configs/strategy.yaml`）
  - 回测记录：最近一次回测的JSON结果（若无，则不包含）
- **数据库记录**：向 PostgreSQL `strategy_backups` 表插入记录，包含策略ID、备份路径、时间戳、版本号等。
- **版本回滚**：支持通过 `strategy_backups` 表查询历史版本，并提供回滚接口（见第9节）。

### 3.4 回测引擎对接
- **接口封装**：编写 `internal/backtest/service.go`，封装本地回测引擎调用。
- **回测数据**：默认使用最近90天数据，可通过配置调整。
- **结果校验**：回测完成后，对比新旧策略的胜率、最大回撤、夏普比率、最大连续亏损。**达标条件**：胜率提升≥5% 且 最大回撤降低≥10%。
- **报告生成**：输出回测报告（JSON格式），包含所有关键指标，并保存到备份目录中。

### 3.5 自动部署开关控制
- **配置项**：环境变量 `AUTO_DEPLOY_ENABLED`，默认为 `false`。
- **部署前二次校验**：若开关开启，且回测达标，则先在测试网运行1小时，监控异常（如订单失败、API限频）。无异常后方可推实盘。
- **部署日志**：实盘部署记录写入 `deploy_logs` 表，包含策略ID、版本、部署时间、部署结果。
- **人工确认**：若开关关闭，则生成优化报告通过消息发送给我，等待我确认后再部署。

## 3.6 回测验证规范（引用 Backtest Expert）
当进行任何策略回测时，必须遵循以下原则：
- **目标**：寻找“最不易失效”的策略，而非纸面收益最高的策略。
- **参数测试**：必须测试参数范围的多个值（如基准值的 50%、100%、150%），寻找性能稳定的平台区，避免仅依赖最优值。
- **摩擦模拟**：滑点按典型值的 1.5-2 倍设置，模拟最差成交价；考虑部分成交、订单拒绝等现实因素。
- **时间维度**：按年分析表现，要求策略在大多数年份盈利，不依赖特定年份。
- **样本量**：至少 30 笔交易，低于此数视为统计不可靠。
- **失败诊断**：若回测结果异常（如胜率>90%、回撤极小），优先怀疑前瞻偏差或数据错误。
- **参考文档**：回测过程中遇到疑问，可查阅 `backtest-expert` 技能的 `references/methodology.md` 和 `references/failed_tests.md`。

## 5. 监控与告警

### 5.1 策略监控指标埋点
- **接入Prometheus**：在 `cmd/strategy/main.go` 中暴露 `/metrics` 端点。
- **新增指标**：
- `strategy_win_rate{id="xxx"}` Gauge：实时胜率。
- `strategy_optimize_count` Counter：优化触发次数。
- `backtest_duration_seconds` Histogram：回测耗时。
- **Grafana面板**：可自行配置可视化，如有需要可请求我协助。

### 5.2 异常告警触发条件
- **告警级别**：
- **P0（紧急）**：策略备份失败、API密钥失效、部署失败 → 立即通知（邮件+终端）。
- **P1（高）**：胜率连续3天＜60% → 立即通知。
- **P2（中）**：回测结果回撤＞5%、回测耗时过长 → 每日简报中提醒。
- **告警方式**：邮件（通过技能 `himalaya` 或自定义邮件技能） + 终端通知（通过 `notify-send` 或 `say`）。

## 7. 多端同步预留设计

### 7.1 API设计预留
- **内部通信**：采用gRPC，接口定义放在 `api/proto/`，生成Go/Java/Flutter客户端代码。
- **外部API**：RESTful API，通过gRPC-gateway转换，供前端管理面板使用。
- **幂等性**：所有写操作接口需支持 `request_id` 参数，保证重复请求不会产生副作用。

### 7.2 数据格式统一
- **策略参数/回测结果**：使用Protobuf定义，例如：
```protobuf
message StrategyParams {
  string strategy_id = 1;
  map<string, string> params = 2;
  double win_rate = 3;
}
跨语言兼容：确保Protobuf定义向后兼容，字段编号不可复用。
## 8. 测试体系完善

### 8.1 单元测试覆盖
- **核心函数**必须编写单元测试，位于 `*_test.go` 文件中，与源代码同目录。
- **测试要求**：
  - `calc_win_rate_test.go`：测试不同输入下的胜率计算，边界情况（0 笔交易、30 笔交易）。
  - `backup_strategy_test.go`：测试备份目录创建、文件复制、数据库记录插入。
  - `backtest_validate_test.go`：测试回测结果达标判断逻辑。
- **覆盖率目标**：核心包（internal/）覆盖率 ≥80%。

### 8.2 集成测试场景
- **策略优化全流程测试**：
  - 模拟数据：向 `trades` 表插入 30 笔交易，胜率 65%。
  - 触发优化流程，检查是否触发冷静期、备份、回测、部署（若开启）。
  - 验证数据库记录、备份文件、日志。
- **测试数据**：使用测试数据库，避免污染生产环境。

## 9. 风险兜底机制

### 9.1 策略回滚机制
- **回滚窗口**：新策略部署后 24 小时内可回滚。
- **回滚接口**：`POST /api/v1/strategy/rollback/{id}/{version}`
- **回滚前校验**：自动比较旧策略与当前策略的胜率（基于最近 30 笔），若旧策略胜率更高，则允许回滚；否则提示风险，需人工确认。
- **回滚操作**：从备份目录恢复代码和配置，重启策略服务。

### 9.2 极端场景防护
- **参数修改限制**：单次优化最多修改 2 个核心参数（如网格间距、止损比例），避免过度优化。
- **黑天鹅模拟**：回测时增加一个“黑天鹅”场景：模拟市场暴跌 10%，要求策略最大回撤仍 ≤5%。
- **测试网预跑**：任何实盘部署前，必须在测试网运行 1 小时，监控异常（如订单失败、API 限频、panic）。

## 13. 版本控制与文档同步

### 13.1 代码修改前置条件
每次对项目代码进行任何修改（包括策略优化、功能开发、重构、Bug 修复）之前，必须严格执行以下流程：
1. **提交当前工作区**：确保当前分支的所有改动都已提交（commit）到本地 Git 仓库。若存在未完成的修改，需先完成一个合理的阶段性提交。
2. **推送至远程仓库**：运行 `git push origin <当前分支>`，将本地提交推送到 GitHub 远程仓库。
3. **确认推送成功**：检查终端输出，确保推送成功（无错误）。若推送失败，需解决网络或认证问题后再继续。
4. **记录版本号**：在开始修改前，记录当前的提交哈希或版本标签（如 `git describe --tags` 或自定义版本号 `vYYYYMMDD-HHMM`），并通知我：“当前代码已备份至 GitHub，版本号为 `xxxx`，现在开始修改。”
5. **修改完成后**：完成代码修改后，再次提交并推送，并附上清晰的 Commit 信息，包含新版本号（如果适用）。同时，必须提供一份**文档更新说明**，指导我如何同步更新相关的 `.md` 文件（如 `SOUL.md`、`USER.md`、`AGENTS.md` 或其他技能文档），例如：
 - 新增了哪些功能，需要在文档中补充说明；
 - 修改了策略参数，需要更新 `USER.md` 中的策略配置部分；
 - 更新了技能，需要更新 `SKILL.md` 或技能列表。

### 13.2 文档修改同步
当你需要修改任何 `.md` 文件（如本文档本身）时，同样遵循上述前置条件：修改前确保当前版本已推送，修改后告知我修改了哪些内容，以及如何应用这些修改（如果是配置文件，可能需要我手动合并或重启服务）。

### 13.3 版本号规范
- 对于策略优化相关的改动，建议使用**语义化版本号**（如 `v1.2.3`）或**日期时间戳**（如 `20260310-1430`）。
- 每次推送的 Commit 信息必须包含版本号（如 `feat(strategy): 优化网格参数，版本 v1.2.4`），以便追溯。

### 13.4 沟通示例
修改完成后，你可以这样通知我：
代码修改已完成，已推送至 GitHub，新版本号 v1.2.5。
主要改动：

调整了网格间距参数（从 0.5% 改为 0.3%）

优化了胜率计算缓存逻辑

请同步更新以下文档：

USER.md 中“策略配置”部分的胜率计算说明需补充缓存描述。

AGENTS.md 的 3.1 节已包含缓存说明，无需修改。

需要我帮你更新这些文档吗？


### 13.5 紧急修改的例外处理
对于紧急修复（如 P0 级生产环境 Bug），可以适当简化流程以快速响应，但必须保证可追溯：
- 允许先在分支上修改并验证，然后立即推送，Commit 信息中必须注明 `hotfix: 紧急修复 xxx，版本 v1.2.5-hotfix1`。
- 推送后应立即通知我，说明紧急情况、修改内容、版本号。
- 事后需补全文档更新（如有必要）并补齐常规流程中的步骤（如更新测试、完善文档）。
